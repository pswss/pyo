# Step 04 — 실행 방법과 배포: run.py vs compiled.py

## 학습 목표
- run.py와 compiled.py의 차이와 사용 시점을 안다
- stickytape가 무엇을 하는지 이해한다
- 대회 제출 전 체크리스트를 숙지한다

---

## 1. 왜 두 가지 실행 방법이 있는가? (10분)

### Erebus의 제약

Erebus 시뮬레이터는 로봇 컨트롤러로 **파일 1개만** 받습니다.

```
우리 프로젝트 src/ 구조:
  src/
  ├── main.py
  ├── robot/robot.py
  ├── mapping/mapper.py
  ├── agent/agent.py
  └── ... (총 46개 파일)

Erebus에 올릴 수 있는 것:
  파일 1개 뿐!
```

로컬 PC에서는 src/ 폴더 전체가 있으니 `run.py`가 `import main`을 통해
나머지 46개 파일을 모두 불러올 수 있습니다.

하지만 **대회장 심판 PC에는 src/ 폴더가 없습니다.**
파일 한 개만 제출하는 규정이기 때문입니다.

## 2. 두 파일의 차이 한눈에 보기 (10분)

| 항목 | **run.py** | **compiled.py** |
|------|-----------|----------------|
| 파일 크기 | 7줄 | 약 225,000자 (225KB) |
| 코드 위치 | src/ 폴더를 직접 참조 | 모든 코드가 내부에 포함 |
| 수정 후 반영 | 저장하면 바로 적용 | 재컴파일 필요 |
| 사용 시점 | **개발 / 테스트** | **대회 제출** |
| src/ 없으면? | 오류 (import 실패) | 정상 실행 |
| 수정 권장 | 자유롭게 수정 | **직접 수정 금지** |

### run.py 내용 (전체)

```python
import sys
import os

# src/ 폴더를 자동으로 찾아서 경로에 추가
src_dir = os.path.dirname(os.path.abspath(__file__))
sys.path.insert(0, src_dir)

import main
```

단 7줄. src/ 폴더를 Python 경로에 추가하고 main.py를 실행합니다.

### compiled.py가 실행되는 원리

```
compiled.py 실행
    │
    ▼
임시 폴더 생성 (C:\Users\...\AppData\Local\Temp\tmpXXXXX\)
    │
    ▼
내부에 저장된 각 모듈을 임시 폴더에 파일로 풀어냄
  tmpXXXXX/main.py
  tmpXXXXX/robot/robot.py
  tmpXXXXX/mapping/mapper.py
  ...
    │
    ▼
임시 폴더를 Python 경로에 추가하고 main() 실행
    │
    ▼
시뮬레이션 종료 시 임시 폴더 자동 삭제
```

오류 메시지에서 이런 경로가 보이는 것이 바로 이 때문입니다:
```
File "C:\Users\snowbot\AppData\Local\Temp\tmpk9mfano9\executor\executor.py"
```
실제 파일이 여기 있는 게 아니라, 실행할 때 임시로 풀린 것입니다.

## 3. stickytape — 번들링 도구 (10분)

stickytape는 여러 Python 파일을 하나로 합쳐주는 도구입니다.

### 설치 확인
```bash
C:\Users\snowbot\AppData\Roaming\Python\Python314\Scripts\stickytape.exe --version
```

### 컴파일 명령 (PowerShell)
```powershell
# rescate_laberinto-master/ 폴더에서 실행
C:\Users\snowbot\AppData\Roaming\Python\Python314\Scripts\stickytape.exe `
    src/main.py `
    --add-python-path src/ `
    | Out-File -Encoding utf8 compiled_new.py
```

### 명령어 분석

| 부분 | 의미 |
|------|------|
| `stickytape.exe` | 번들링 도구 실행 |
| `src/main.py` | 진입점 (여기서부터 시작) |
| `--add-python-path src/` | src/ 안의 모듈들을 찾을 수 있게 경로 추가 |
| `\| Out-File -Encoding utf8` | 출력을 파일로 저장 (UTF-8 인코딩) |
| `compiled_new.py` | 출력 파일 이름 (테스트용으로 new 붙임) |

### 주의: 재컴파일 시 점수가 달라질 수 있다

> **중요**: 현재 대회용 `compiled.py`는 원저자가 더 완성된 버전의 소스코드로
> 컴파일한 파일입니다. 현재 `src/` 폴더의 코드와 완전히 동일하지 않습니다.
>
> 함부로 재컴파일하면 점수가 낮아질 수 있습니다.
>
> **개발 흐름 권장:**
> 1. `src/`에서 수정 → `run.py`로 테스트
> 2. 충분히 검증된 후에만 재컴파일

## 4. Erebus에서 컨트롤러 파일 전환하기 (10분)

### run.py 사용 (개발 중)

```
Erebus 실행
  → 시뮬레이션 월드 열기
  → 로봇 선택 (DEF ROBOT0)
  → 컨트롤러 필드에서 파일 선택
```

장점: src/ 코드를 수정하고 시뮬레이션을 재시작하면 바로 반영됩니다.

### compiled.py 사용 (대회 / 최종 테스트)

```
Erebus 실행
  → 시뮬레이션 월드 열기
  → 로봇 선택 (DEF ROBOT0)
  → 컨트롤러 필드에서 파일 선택
  → compiled.py 선택
  → 시뮬레이션 시작
```

## 5. 원저자 run.py의 문제와 수정 (5분)

원본 run.py는 원저자 Linux PC의 절대 경로가 하드코딩되어 있었습니다:

```python
# 원본 (문제 있음)
absuloute_dir = r'/home/iitaadmin/simulated_rescue_maze/rescate_laberinto/src'
sys.path.append(absuloute_dir)
```

다른 PC에서 실행하면 경로가 없어서 즉시 오류가 납니다.

```python
# 수정 후 (어느 PC에서도 동작)
src_dir = os.path.dirname(os.path.abspath(__file__))
sys.path.insert(0, src_dir)
```

`__file__`은 현재 실행 중인 파일(run.py)의 경로를 자동으로 반환합니다.
PC가 달라져도 항상 run.py가 있는 폴더(= src/)를 정확히 찾습니다.

## 6. compiled.py 직접 수정이 필요한 경우 (5분)

재컴파일 없이 compiled.py의 버그를 직접 패치해야 할 때가 있습니다.

### 실제 사례: uint8 오버플로 버그

compiled.py 내부에는 각 모듈의 소스코드가 바이트 문자열로 저장되어 있습니다:

```python
__stickytape_write_module('algorithms/np_bool_array/efficient_a_star.py', b'...')
```

이 안에서 특정 텍스트를 찾아 수정할 수 있습니다:

```python
# 수정 전 (버그)
return preference_grid[position[0], position[1]]

# 수정 후 (패치)
return int(preference_grid[position[0], position[1]])
```

단, 이 방법은 수정 범위를 정확히 알고 있을 때만 사용합니다.

```
  □ USB 또는 이메일로 compiled.py 단일 파일 제출
```
