# Step 02 — 시뮬레이션 루프와 실행 흐름

## 학습 목표
- Webots/Erebus 시뮬레이션이 어떻게 돌아가는지 이해한다
- `do_loop()` → `update()` → `run()` 순서를 안다
- 상태기계(StateMachine)가 왜 필요한지 이해한다

---

## 1. Webots 시뮬레이션이란? (10분)

### 실제 로봇 vs 시뮬레이션 로봇

| 항목 | 실제 로봇 | Webots 시뮬레이션 |
|------|----------|-----------------|
| 물리 엔진 | 실제 물리법칙 | 가상 물리 계산 |
| 센서 | 실제 하드웨어 | 가상 센서 (노이즈 포함) |
| 시간 | 실제 시간 | time_step 단위로 진행 |
| 연습 | 로봇 필요 | PC만 있으면 됨 |

### time_step이란?

```
time_step = 32  →  매 32밀리초(ms)마다 시뮬레이션 1 스텝 전진

1초 = 1000ms ÷ 32ms = 31.25번 실행
8분 = 480초 × 31.25 = 15,000번 실행
```

> **비유**: 영화 필름처럼 연속된 사진을 빠르게 보여주는 것.
> 각 "사진"이 한 time_step 입니다.

## 2. 메인 루프 코드 읽기 (15분)

[src/executor/executor.py](../../../src/executor/executor.py) — `run()` 함수

```python
def run(self):
    while self.robot.do_loop():          # ← 시뮬레이션 계속 진행?
        self.robot.update()              # 1. 센서 값 읽기
        self.delay_manager.update(...)   # 2. 타이머 갱신
        self.stuck_detector.update(...)  # 3. 갇힘 감지
        self.do_mapping()               # 4. 지도 업데이트
        self.check_map_sending()        # 5. 시간 종료 체크
        self.state_machine.run()        # 6. 현재 상태 실행
```

### `do_loop()`가 하는 일

Webots가 제공하는 함수. 내부에서:
1. 물리 엔진을 32ms 진행시킴
2. 센서 데이터를 최신화함
3. 시뮬레이션이 계속되면 `True` 반환
4. 일시정지/종료면 `False` 반환 → while 루프 탈출

## 3. 상태기계가 필요한 이유 (15분)

### 상태기계가 없다면?

```python
# 상태기계 없이 작성한다면...
def run_without_state_machine():
    while robot.do_loop():
        if victim_found and not reporting:
            start_reporting = True
        
        if start_reporting:
            if not stopped:
                stop_wheels()
                stopped = True
            elif not rotated:
                rotate_to_victim()
                rotated = True
            elif not sent_signal:
                send_signal()
                sent_signal = True
            elif waiting_time < 3.0:
                waiting_time += 0.032
            else:
                start_reporting = False
                stopped = False
                rotated = False
                sent_signal = False
                waiting_time = 0
        
        elif exploring and not victim_found:
            # ... 탐색 코드
        
        elif going_to_start:
            # ... 복귀 코드
```

→ 변수가 폭발적으로 늘어남, 어디서 버그 났는지 추적 불가!

### 상태기계 사용 후

```python
# 상태기계 사용
state_machine.create_state("explore",         state_explore,         {"report_fixture", "stuck", "end"})
state_machine.create_state("report_fixture",  state_report_fixture,  {"explore", "send_map"})
state_machine.create_state("stuck",           state_stuck,           {"explore", "end"})
state_machine.create_state("end",             state_end)

# 메인 루프: 딱 한 줄!
state_machine.run()
```

→ 각 상태가 독립적, 전환 규칙이 명확, 디버그 쉬움!

## 4. 6가지 상태 완전 분석 (15분)

[src/executor/executor.py](../../../src/executor/executor.py)

```
┌──────────────────────────────────────────────────────────┐
│                    상태 전환 다이어그램                    │
│                                                          │
│   시작                                                   │
│    │                                                     │
│    ▼                                                     │
│  [init] ──────────────────────────────→ [explore]        │
│  초기화/보정                              탐색 중          │
│                                           │  ↑           │
│                              피해자발견  ↙  ↖ 보고완료    │
│                                [report_fixture]          │
│                                피해자 신고                │
│                                                          │
│  [explore] ──시간거의종료──→ [send_map] ──→ [explore]    │
│                                                          │
│  [explore] ──갇힘감지──→ [stuck] ──탈출──→ [explore]    │
│                                                          │
│  [explore/stuck] ──임무완료──→ [end]                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 각 상태 설명

| 상태 | 하는 일 | 전환 조건 |
|------|--------|----------|
| `init` | GPS/자이로 보정, 360° 회전 스캔 | 보정 완료 → explore |
| `explore` | A*로 목표 좌표 이동 + 피해자 감지 | 피해자 발견 → report_fixture |
| `report_fixture` | 피해자 방향 정렬 → 신호 전송 | 신고 완료 → explore |
| `send_map` | 현재 지도 심판 시스템 전송 | 전송 완료 → explore |
| `stuck` | 후진 + 회전으로 탈출 시도 | 탈출 성공 → explore |
| `end` | 최종 지도 전송 + 종료 신호 | (종료) |

## 5. Sequencer — 비동기 프로그래밍 (10분)

피해자 신고 과정은 여러 단계가 순서대로 실행되어야 합니다:

```
1. 바퀴 정지 (즉시)
2. 피해자 방향으로 회전 (1~2초 걸림)
3. 정밀 정렬 (0.5초 걸림)
4. 1.5초 대기
5. 신호 전송 (즉시)
6. 후진 (0.1초)
```

매 32ms마다 `state_report_fixture()`가 호출되는데,
어떻게 "1.5초 대기"를 구현할까요?

```python
# Sequencer가 없다면
timer = 0
def state_report_fixture():
    global timer
    if step == 1:
        stop_wheels()
    elif step == 2 and rotation_done:
        ...
    # 단계 추적이 매우 복잡해짐

# Sequencer 사용
def state_report_fixture(change_state):
    self.sequencer.start_sequence()
    self.seq_move_wheels(0, 0)           # 1. 정지
    self.seq_rotate_to_angle(angle)      # 2. 회전 (완료될 때까지 대기)
    self.seq_delay_seconds(1.5)          # 3. 1.5초 대기
    self.seq_send_victim_signal()        # 4. 신호 전송
    self.sequencer.seq_reset_sequence()  # 처음부터 반복
```

> **핵심**: `seq_*` 함수들은 한 번에 실행되지 않습니다.
> 각각 "이번 스텝에 여기까지 왔나?"를 체크하며 순서대로 실행됩니다.
